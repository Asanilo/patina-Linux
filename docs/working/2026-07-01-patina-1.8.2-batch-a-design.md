# Patina 1.8.2 Batch A Design

Date: 2026-07-01

Status: approved for implementation planning

## 1. Objective

Prepare an independently releasable `1.8.2` batch that improves three trust-sensitive Linux paths:

1. Make local API port changes atomic and make API Token handling suitable for a local authenticated interface.
2. Selectively port upstream `1.8.1` custom-category rename, merge, and excluded-filter behavior without importing its built-in-category risks.
3. Expand Linux browser-family matching while preserving the existing signed Firefox/Zen extension path and URL privacy behavior.

This batch does not include the later Data/History resource work, Linux storage migration, timezone additions, or Tools write-side API/MCP work.

## 2. Product Boundaries

- Keep Patina Linux-first, local-first, and GNOME Wayland-first.
- Do not add cloud services, accounts, built-in AI chat, or remote configuration.
- Do not change the existing web activity URL privacy modes.
- Keep the current signed Firefox extension and Manifest V2 compatibility path.
- Keep Windows platform sources as compatibility code; do not expand Windows release scope.
- Release Batch A independently as `1.8.2` after the complete validation bar passes.

## 3. Local API Configuration

### 3.1 Ownership

- `engine/api` owns listener preparation, active listener state, and runtime Token state.
- `data/repositories` owns the persisted API port and one-time legacy Token migration state.
- `commands/settings` remains a thin Tauri boundary over a dedicated API configuration service.
- `features/settings` owns the user interaction and status presentation.

The local API fields must no longer depend on the generic Settings save path for runtime changes.

### 3.2 Atomic Port Change

Changing the port follows this sequence:

1. Validate the requested port.
2. Bind a new `TcpListener` while the existing listener remains active.
3. Prepare any accompanying persisted state.
4. Install the pre-bound listener as the active runtime.
5. Signal the previous listener to stop.

If validation, binding, or persistence fails, the existing listener and persisted configuration remain unchanged. A bind conflict must be returned to the UI as an actionable error. Applying the currently active port is a no-op.

The runtime snapshot distinguishes configured state from confirmed listening state so diagnostics cannot report an intended port as active before binding succeeds.

### 3.3 Token Generation And Storage

- The `api_token` file is the only long-term Token source of truth; SQLite no longer stores or updates the active Token.
- On upgrade, a legacy SQLite Token may seed the Token file only when the file is missing, after which runtime and Settings reads use the file.
- Generate Tokens from the operating system random source, not timestamps.
- Preserve the existing `patina_api_` prefix for recognizability.
- Write through a temporary file in the Token directory and atomically rename it.
- On Unix, create and repair the Token file with mode `0600`.
- Never print the Token in startup or error logs.
- Never include the Token in diagnostics responses.
- A Token-only rotation updates authentication without restarting the listener.

Token rotation is explicit. The Settings action explains that existing browser extension and MCP credentials will stop working, rotates the Token, refreshes local state, and copies the complete browser configuration. Patina does not attempt to write another application's extension storage.

### 3.4 Failure Semantics

- Port bind failure: preserve the old listener, port, Token, and database values.
- Token file failure: preserve the old Token and listener.
- Port persistence failure: do not install the prepared listener.
- Clipboard failure after successful rotation: keep the successful rotation and show a copy-specific error; do not roll back security state.
- Restart recovery: load persisted settings, enforce Token file permissions, and report listener failure through diagnostics without exposing the Token.

## 4. Classification Management

### 4.1 Supported Operations

- User-created custom categories may be renamed.
- A custom category may be merged into another custom category or an assignable built-in category.
- Built-in categories may still change color and receive app/domain assignments.
- Built-in categories cannot be renamed, merged away, or deleted.
- `system` remains non-assignable and `other` remains the unclassified fallback.

### 4.2 Name Identity

Category-name identity uses a shared comparison key:

1. trim and collapse whitespace;
2. apply Unicode compatibility normalization;
3. apply locale-independent case folding suitable for identifiers.

Creation rejects a name whose comparison key matches any visible built-in or custom category. Renaming a category may change capitalization when its own comparison key stays the same. Renaming to another category's key requires an explicit merge confirmation.

### 4.3 Merge Behavior

Merging rewrites both app overrides and web-domain overrides from the source custom category to the target. The target keeps its existing color; otherwise it inherits the source color. The source label, color, and persisted custom ID are removed only in the draft state. Existing classification transaction behavior persists the complete plan before updating the runtime mapper.

### 4.4 Excluded Filter

Classification gains a separate `excluded` filter:

- an app is excluded when its effective tracking state is disabled;
- a web domain is excluded when its recording state is disabled;
- normal filters do not silently mix excluded candidates into the unclassified group.

The UI follows existing Quiet Pro controls and does not add a new page or nested card structure.

## 5. Linux Browser Families

### 5.1 Rust Matching

`domain/web_activity` remains the single policy owner for supported foreground browser executables. Matching is case-insensitive and grouped by Chromium or Firefox family.

The Linux candidate set includes tested aliases for:

- Chromium, Google Chrome, Brave, Microsoft Edge, Opera, Vivaldi, Thorium;
- Firefox, Zen, Floorp, Iceweasel, LibreWolf.

Windows aliases already retained by compatibility code are not treated as Linux packaging evidence. Each Linux alias must have a focused resolver test.

### 5.2 Extension Identity

The Firefox extension checks specific forks before generic Firefox so a user agent containing both values does not collapse Zen into Firefox. When available, browser runtime metadata may refine the result; user-agent detection remains the compatibility fallback.

Browser identity affects diagnostics and recorded browser metadata only. It does not weaken the foreground-browser gate, change incognito rejection, or change URL privacy behavior.

### 5.3 Unsupported Browsers

Unsupported foreground executables do not create web activity. Diagnostics should expose an unsupported/mismatch reason where the existing snapshot contract can do so without creating a parallel installation wizard.

## 6. Settings Interaction

The local API settings surface provides:

- a validated port input with an independent Apply action;
- a masked, read-only Token value;
- show/hide and copy actions;
- an explicit Rotate Token and Copy Configuration action with confirmation;
- listening, bind-failure, Token-present, and copy-failure feedback using existing semantic status styles.

Classification adds rename and delete actions only for custom categories. A duplicate rename presents a merge confirmation. The excluded filter appears alongside the existing classification filters.

No new visual direction is introduced; implementation follows Quiet Pro tokens and existing control archetypes.

## 7. Validation

### 7.1 Automated Tests

- API server test: a conflicting target port leaves the old listener reachable.
- API server test: a successful change activates the new port and closes the old port.
- Auth test: generated Tokens are not deterministic timestamp values.
- Unix auth test: newly created and repaired Token files are `0600`.
- Logging/static check: startup paths do not print Token contents.
- Settings tests: port apply and Token rotation report partial failures correctly.
- Classification tests: case-insensitive duplicate rejection, case-only rename, custom-to-custom merge, custom-to-built-in merge, app/domain rewrite, and built-in operation restrictions.
- Browser tests: every supported Linux alias resolves to the expected family; Zen-specific identity wins over Firefox fallback.
- Existing extension checks, API schema tests, and release policy tests remain green.

### 7.2 Manual Acceptance

1. Start Patina and verify the current local API health endpoint.
2. Occupy a candidate port and apply it; verify the old endpoint remains reachable and Settings shows the bind error.
3. Apply a free port; verify the new endpoint responds and the old endpoint closes.
4. Rotate the Token; verify the Token file mode, copied configuration, old credential rejection, and new credential success.
5. Create `Focus`; verify `focus` is rejected and a case-only rename to `FOCUS` succeeds.
6. Merge a custom category containing app and web rules; verify both move and statistics remain readable.
7. Verify built-in category controls do not expose rename/delete.
8. Verify Zen web activity still records and diagnostics identify Zen.

### 7.3 Release Gate

At minimum run:

- `npm test`
- `npm run test:replay`
- `npm run test:classification`
- `npm run test:settings`
- `npm run build`
- `cargo test --manifest-path src-tauri/Cargo.toml`
- extension checks
- repository release checks for `1.8.2`

Version files, changelog, tag, updater metadata, `.deb`, and AppImage assets must remain consistent with the Linux-only release policy.

## 8. Out Of Scope For 1.8.2

- Data/History and background resource optimization.
- Linux data directory or WebView cache migration.
- API timezone response additions.
- Tools write-side HTTP API or MCP tools.
- KDE or wlroots foreground-window support.
- Browser extension automatic update infrastructure.
